# Feature Plan: F2 Profile With Settings Editing

## Scope

Implement the authenticated profile experience so `/profile` provides one stable surface for:

- user identity overview,
- editable settings backed by `/api/settings`,
- learning analytics from `/api/stats/overview`,
- session history from `/api/history` with optional detail expansion.

This feature is limited to the profile page and its supporting profile API/types/utilities.
It does not change auth routing, backend contracts, or chat/cases flows outside profile-linked
navigation and display-name usage.

## Goal

Ship a profile page that lets the user:

- view `displayName` and read-only `username`,
- edit and save `displayName`,
- preserve unknown backend `settings` keys during updates,
- review aggregate training stats,
- inspect session history without losing the main profile context.

- `src/views/ProfileView.vue` already loads settings, stats, and history on mount.
- `src/composables/useProfileApi.ts` already exposes:

  - `stats()`

- `src/utils/mergeSettings.ts` already preserves unknown settings keys through a spread merge.
- `src/types/index.ts` already defines:

Remaining `F2` work is therefore mostly about locking the feature contract, verifying edge cases,
and tightening any gaps between current implementation and the roadmap/spec.

## Locked Decisions

### Profile Data Sources

- Settings come from `GET /api/settings`.
- Stats come from `GET /api/stats/overview`.
- Session history comes from `GET /api/history`.
- History detail comes from `GET /api/history/{sessionId}` when the user expands an item.

No extra aggregation endpoint is introduced for the profile page.

### Settings Ownership

- The editable profile form uses the backend `UserSettings` payload as the source of truth.
- MVP editing scope is limited to `displayName`.
- Unknown keys inside `settings: Record<string, unknown>` must survive every save.
- Preservation happens in one canonical merge path, not by ad hoc spreading inside the view.

### Save Behavior

- Profile edits are persisted through `useProfileApi().updateSettings(...)`.
- Submit must be disabled while the save request is pending.
- Success and failure feedback must be shown inline on the profile page.
- Save UX must remain local and non-modal.

### History UX

- Session history stays on the profile page as a secondary section.
- Detail expansion is lazy-loaded per session and cached after first fetch.
- History failures for a single item must not break the whole profile page.

## Tasks

### F2.1 Implement Profile Data Fetch

Ensure the profile page loads all required authenticated data in a predictable way.

#### Required behavior

- `ProfileView.vue` loads on mount:
  - `profileApi.settings()`
  - `profileApi.stats()`
  - `profileApi.history()`
- Initial settings, stats, and history requests run in parallel.
- Blocking load failure shows one clear profile-level error state.
- Successful load hydrates:
  - identity summary,
  - editable form state,
  - stats panel,
  - history list.

#### Rules

- Keep initial loading explicit with skeleton or equivalent placeholder states.
- Do not split the initial profile fetch across unrelated child components.
- Do not duplicate profile-fetch logic in layout or store layers.

Goal: `/profile` has one clear initial-load path and one understandable failure mode.

---

### F2.2 Implement Settings Update Flow

Persist display-name edits safely while preserving the full backend settings payload.

#### Required behavior

- Form edits `displayName` only.
- Save action calls `useProfileApi().updateSettings(...)`.
- `updateSettings()` preserves unknown settings keys through the canonical merge strategy:
  - fetch current settings if needed,
  - merge existing `current.settings` with updated keys,
  - send the merged payload via `PUT /api/settings`.
- Returned settings replace local page state after a successful save.

#### Typed contract requirements

- `UserSettings.settings` remains `Record<string, unknown>`.
- `UpdateUserSettingsRequest.settings` remains `Record<string, unknown>`.
- No `any` is introduced in the view, composable, or merge utility.

#### Sync considerations

- If profile changes affect user-facing identity outside the page, the canonical auth display-name
  source must stay consistent or be refreshed in a defined follow-up path.
- `ProfileView.vue` may continue to render freshly returned settings immediately even if the
  auth store display name updates later.

Goal: display-name editing is safe, typed, and cannot silently drop unknown backend fields.

---

### F2.3 Add Save UX And Validation States

Make the profile form deliberate under pending, success, and failure conditions.

#### Required behavior

- Save button is disabled while request is pending.
- Save button exposes loading state while request is pending.
- Save success is shown inline after a successful update.
- Save failure is shown inline after an unsuccessful update.
- Duplicate submits are prevented while pending.

#### Validation expectations

- Display name should be trimmed before submit.
- Empty/invalid values should not produce ambiguous UX.
- Error messaging should stay local to the settings card, not global to the entire page.

#### Optional tightening

- If useful, treat unchanged trimmed display name as a no-op and keep the button disabled.
- If the auth store is not updated immediately after save, document that behavior clearly in code
  and avoid showing contradictory identity values on the same screen.

Goal: the settings form behaves predictably and transparently under real request timing.

---

### F2.4 History Detail Expansion And Resilience

Keep history useful without turning profile into a second full session page.

#### Required behavior

- History list renders compact session summaries.
- Expanding a history item fetches detail only once per `sessionId`.
- Expanded detail is cached locally after the first successful fetch.
- Failure to load one detail item remains non-blocking for the rest of the page.

#### Rules

- Keep history detail secondary to settings and stats.
- Do not navigate away from `/profile` just to inspect one history item in MVP.
- Avoid refetching the same detail payload on every expand/collapse cycle.

Goal: history inspection is informative, fast on repeat open, and isolated from page-level failure.

## Definition Of Done

- `/profile` loads settings, stats, and history from the correct authenticated endpoints.
- `ProfileView.vue` shows one explicit loading state and one explicit blocking-load error state.
- The page renders:
  - identity summary,
  - editable display-name form,
  - learning stats,
  - session history.
- Saving settings calls `useProfileApi().updateSettings(...)` only.
- Unknown backend `settings` keys are preserved on every save through the canonical merge path.
- Save button is disabled/loading while pending.
- Inline success and error feedback is present for save attempts.
- History detail expansion is lazy, cached, and non-blocking on per-item failure.
- No `any` is introduced.
- ASCII-only source is preserved.

## Risks And Mitigations

**Risk: profile save drops unknown backend settings fields.**  
Mitigation: keep every update routed through `useProfileApi().updateSettings()` and `mergeSettings()`;
never construct a partial `settings` payload directly in the view.

**Risk: page-level load failure hides which source broke.**  
Mitigation: keep one blocking initial-load error for MVP simplicity, but isolate history-detail
fetch failures from the main load path.

**Risk: saved display name diverges from auth-shell identity.**  
Mitigation: define one follow-up sync strategy for `authGate` and avoid showing contradictory names
in the same render path.

**Risk: history detail expansion causes repeated unnecessary requests.**  
Mitigation: cache detail by `sessionId` after first successful fetch and reuse it on later expands.

## Execution Order

1. Verify `useProfileApi` and types match the locked `F2` contract for settings, stats, history, and merge safety.
2. Tighten `ProfileView.vue` initial-load behavior and form-state handling if it still deviates from the plan.
3. Validate save UX states and display-name synchronization behavior.
4. Validate lazy history-detail expansion, caching, and failure isolation.
5. Run type/lint checks and manual profile-flow verification.
