# Feature Plan: F3 Simulation Streaming And Recovery

## Scope

Implement the simulation streaming layer so chat sessions render opening and patient replies as
server-pushed progressive content while keeping REST as the canonical source of truth for session
state, recovery, and mutation commands.

This feature is limited to the chat streaming lifecycle and its supporting composables, types,
and page orchestration. It does not change backend contracts, case selection flow, or diagnosis
scoring rules outside the states that depend on streaming.

## Goal

Ship a chat streaming experience that:

- renders opening generation progressively,
- renders patient replies progressively after doctor messages,
- survives page refresh during in-flight generation,
- surfaces actionable recovery states for stream errors and failed openings,
- keeps all socket/STOMP details hidden behind a dedicated composable.

## Current State

The feature already has a usable baseline, but it is only partially aligned with the roadmap:

- `src/composables/useSimulationSocket.ts` already exists and encapsulates STOMP/SockJS topic
  subscription behind `connect()`, `disconnect()`, `reconnect()`, and `onChunk()`.
- `src/views/ChatView.vue` already:
  - fetches session state on mount,
  - connects the socket when opening or message streaming is in flight,
  - appends incoming `chunk` payloads,
  - reloads session state on `done`,
  - reloads session state after stream `error`,
  - supports retrying failed opening generation through `retryOpening=true`.
- `src/composables/useSimulationApi.ts` already provides the required REST calls:
  - `getSession()`
  - `sendMessage()`
  - `diagnose()`
- Typed stream payload validation already exists through `StreamChunk` and `isStreamChunk`.

Gaps against the requested `F3` scope:

- streaming lifecycle is still split between the socket composable and `ChatView.vue` instead of
  exposing a more explicit chat-oriented streaming contract,
- refresh recovery is only partially restored through `streamingStatus.inFlight` and does not yet
  clearly separate opening recovery from message recovery in UI state,
- reconnect and stale-session handling are minimal and not yet documented as deterministic,
- socket error handling is generic and does not fully map to the user-facing recovery rules from
  `PLAN.md`,
- the current transport uses STOMP/SockJS directly rather than a VueUse-centered wrapper, so the
  feature plan must preserve the abstraction boundary and avoid leaking transport specifics upward.

## Locked Decisions

### REST Remains Canonical

- `GET /api/simulations/{sessionId}` remains the source of truth for session snapshots.
- `POST /api/simulations/{sessionId}/message` remains the only way to send doctor messages.
- `POST /api/simulations/{sessionId}/diagnose` remains the only way to submit a diagnosis.
- Streaming augments the active timeline only; it does not replace REST revalidation.

### Transport Stays Behind A Composable

- Page components must not own STOMP topics, SockJS URLs, or subscription frame logic.
- `useSimulationSocket` remains the only boundary that knows transport details.
- If the transport internals change later, `ChatView.vue` should keep consuming the same small,
  event-oriented API rather than being rewritten around protocol details.

### Recovery Must Be Stateful

- Opening recovery and patient-reply recovery are not the same thing and must be modeled
  explicitly in UI state.
- Refresh on `/chat/:sessionId` always starts with `getSession(sessionId)` before rebuilding any
  transient streaming UI.
- `409` conflict responses and stream failures must produce contextual recovery messaging, not only
  generic page-level errors.

### Session Route Ownership

- Route param `sessionId` remains the only frontend session identifier.
- No `/chat/active` route is introduced.
- Recovery from refresh, reconnect, and failed stream happens inside the existing chat route flow.

## Tasks

### F3.1 Formalize The Streaming Contract In `useSimulationSocket`

Tighten the composable so it exposes a predictable contract tailored to simulation streaming
instead of only low-level chunk subscription.

#### Required changes

- Keep one explicit socket status model covering:
  - `idle`
  - `connecting`
  - `open`
  - `reconnecting`
  - `closed`
  - `error`
- Expose a clear session-scoped connect lifecycle:
  - connect to one `sessionId`,
  - resubscribe safely when the session changes,
  - disconnect cleanly on route leave,
  - reconnect deterministically when requested.
- Keep validated chunk delivery centralized:
  - parse incoming payloads once,
  - ignore malformed frames safely,
  - dispatch only validated `StreamChunk` values.
- Preserve transport encapsulation:
  - no STOMP topic strings in views,
  - no direct SockJS/STOMP client handling in `ChatView.vue`.

#### Recovery requirements

- Track enough internal state to reconnect to the active session without requiring the view to
  reconstruct topic details.
- Ensure reconnect does not duplicate active subscriptions.
- Ensure transport-level failures set a typed error state consumable by the view.

Goal: the socket composable becomes the single reliable boundary for connection lifecycle and
validated chunk delivery.

---

### F3.2 Separate Opening Streaming From Message Streaming In Chat State

Make `ChatView.vue` model the two stream types explicitly so recovery logic is easier to reason
about and consistent with backend session state.

#### Required behavior

- On initial session load, distinguish:
  - opening generation in progress,
  - patient reply in progress,
  - no active stream.
- Restore the correct transient UI using backend state:
  - `openingStatus` for opening generation,
  - `streamingStatus.inFlight` and `streamingStatus.type` for patient reply recovery.
- Keep opening UI non-interactive until:
  - `openingStatus === "OPENING_READY"`
  - `state === "IN_PROGRESS"`
- Keep message-send UI disabled while a reply is in flight.

#### View-state requirements

- Avoid one undifferentiated `isStreamingActive` flag when the UI needs different behavior for:
  - opening,
  - patient reply,
  - idle state.
- Keep transient state explicit, for example:
  - active stream kind,
  - pending doctor message,
  - accumulated streamed content,
  - last recovery/error message.
- Ensure transient state is cleared after successful REST refresh on stream completion.

Goal: the chat page knows what is streaming and can render and recover accordingly.

---

### F3.3 Implement Deterministic Stream Event Handling

Handle `chunk`, `done`, and `error` events in a way that keeps both the timeline and the canonical
session state synchronized.

#### Required behavior

- `chunk`:
  - append content to the correct in-progress bubble,
  - preserve ordering for progressive render.
- `done`:
  - stop transient pending state,
  - refresh canonical session state from REST,
  - clear temporary stream buffers only after refresh handling completes.
- `error`:
  - stop transient pending state,
  - surface actionable recovery messaging,
  - allow session reload and socket recovery without forcing a route change.

#### Rules

- Do not finalize the timeline exclusively from chunk state; use REST refresh after completion to
  recover authoritative message/session structure.
- Do not leave stale pending UI behind after `done` or `error`.
- Do not let chunk handling silently fail without a visible recovery path in the chat UI.

Goal: stream events produce predictable UI transitions and leave the page in a canonical state.

---

### F3.4 Tighten Refresh And Reconnect Recovery

Complete the refresh-recovery behavior described in `PLAN.md` so the page remains usable after
reloads and transport interruptions.

#### Required behavior

- On `/chat/:sessionId` mount:
  - fetch session state first,
  - decide from the payload whether socket connection is needed,
  - rebuild opening or message-stream transient UI from backend state.
- When the backend reports:
  - `streamingStatus.type === "opening"`: restore opening loader and streamed content target,
  - `streamingStatus.type === "message"`: restore pending patient-reply state,
  - `openingStatus === "OPENING_FAILED"`: show retry CTA using `retryOpening=true`.
- After reconnect or recovered completion:
  - refresh session state from REST when needed,
  - clear stale retry/conflict errors that no longer apply.

#### Failure handling

- Transport reconnect failure must not permanently strand the page in a fake loading state.
- Opening failure must show a specific retry path, not a generic chat error.
- If recovery cannot restore a valid session, the user must still have a clear route back to
  `/cases`.

Goal: refresh and reconnect behave as first-class product flows, not incidental side effects.

---

### F3.5 Align User-Facing Errors With Stateful Backend Rules

Map streaming and conflict failures to the actual simulation lifecycle so users understand what to
do next.

#### Required behavior

- Opening generation failure:
  - show opening-specific failure copy,
  - provide retry action.
- Message-send conflict (`409`):
  - explain that a reply is already in flight or the session is not ready,
  - preserve current session context,
  - refresh/reload when helpful.
- Stream transport failure:
  - explain that live response delivery was interrupted,
  - offer reload/reconnect guidance,
  - avoid presenting it as a missing-session error unless REST load actually fails.
- Diagnosis conflict (`409`) during streaming-sensitive states:
  - show stateful explanation,
  - refresh session state.

#### Rules

- Keep conflict and recovery messaging inline on the chat page.
- Do not reduce lifecycle conflicts to toast-only feedback.
- Do not overwrite a valid loaded session with a blocking fatal error unless the session itself
  truly cannot be fetched.

Goal: backend state rules become understandable chat UX instead of opaque transport failures.

## Definition Of Done

- `useSimulationSocket` exposes a deterministic, session-scoped streaming contract and keeps all
  transport details internal.
- `ChatView.vue` explicitly distinguishes opening streaming, patient-reply streaming, and idle
  state.
- Initial `/chat/:sessionId` load rebuilds the correct transient stream UI from backend session
  payload.
- Stream `chunk`, `done`, and `error` events update the UI and REST revalidation flow
  predictably.
- Opening failure renders a dedicated retry path using `retryOpening=true`.
- Message-send and diagnosis conflicts render contextual inline recovery messaging.
- Refresh during an in-flight opening or patient reply restores the correct pending UI.
- Route leave still disconnects the socket cleanly.
- No `any` is introduced.
- ASCII-only source is preserved.

## Verification

### Manual checks

- Open `/chat/:sessionId` while opening generation is still pending and confirm:
  - the page shows a non-interactive opening state,
  - streamed opening content appears progressively,
  - chat input stays disabled until opening is ready.
- Send a doctor message and confirm:
  - one REST send request is made,
  - patient reply content streams progressively,
  - input remains disabled while the reply is in flight,
  - session state refreshes after stream completion.
- Refresh the page during:
  - opening streaming,
  - patient reply streaming,
  and confirm the correct transient UI is restored after `getSession()`.
- Trigger `OPENING_FAILED` and confirm the retry CTA calls
  `getSession(sessionId, { retryOpening: true })`.
- Trigger a transport failure and confirm the page shows actionable inline recovery messaging
  without losing the loaded session shell.
- Trigger `409` conflicts for send and diagnose actions and confirm the user sees stateful
  guidance rather than a generic error.

## Risks And Mitigations

**Risk: transport details leak into `ChatView.vue` and make future socket changes expensive.**  
Mitigation: keep topic names, client lifecycle, and payload parsing inside `useSimulationSocket`
only; let the view consume a compact event/status API.

**Risk: one generic streaming flag causes incorrect recovery behavior after refresh.**  
Mitigation: model opening and patient-reply streaming separately and rebuild transient UI from
`openingStatus` plus `streamingStatus.type`.

**Risk: stream completion leaves stale pending bubbles or inconsistent session state.**  
Mitigation: always refresh canonical session state from REST after `done`, and clear transient
buffers only after that refresh path settles.

**Risk: transport failure is presented as a fatal page error even when the session still exists.**  
Mitigation: separate session-load failures from live-stream failures and keep recovery messaging
inline when REST state is still available.

## Execution Order

1. Tighten `useSimulationSocket` so its public API fully owns session-scoped connection and chunk
   delivery semantics.
2. Refactor `ChatView.vue` streaming state to distinguish opening recovery from message recovery.
3. Normalize `chunk` / `done` / `error` handling so transient UI and REST refresh happen in one
   predictable sequence.
4. Implement explicit refresh/reconnect recovery and inline stateful error messaging.
5. Run type/lint checks and manual chat-flow verification for opening, message streaming, refresh,
   retry, and conflict scenarios.
