# Feature Plan: F1 Quick Prompts For Chat Buttons

## Scope

Implement the chat quick-action prompts so the three helper buttons above the input:

- are defined from one typed source of truth,
- render from configuration instead of hardcoded button content,
- send their prompt text through the exact same message pipeline as manual doctor messages.

This feature is limited to the chat page and its supporting constants/types. It does not change
the backend contract, the message endpoint, or the streaming lifecycle.

## Goal

Ship three fixed quick prompts for:

- physical examination,
- laboratory diagnostics,
- instrumental diagnostics.

Each button must behave as a shortcut for sending a predefined doctor message via
`POST /api/simulations/{sessionId}/message`.

## Current State

The feature is partially implemented, but not yet aligned with the plan/spec:

- `src/constants/simulationQuickPrompts.ts` already exists, but it only exports a flat array with
  `label` and `content`.
- `src/components/chat/ChatInputBar.vue` already renders quick prompt buttons from props and emits
  `send` with `prompt.content`.
- `src/views/ChatView.vue` already passes `SIMULATION_QUICK_PROMPTS` into `ChatInputBar` and routes
  `@send` to `handleSend()`.

Gaps against the requested `F1` scope:

- prompt config has no stable keys,
- prompt texts are placeholder phrases and do not match the locked detailed templates from `PLAN.md`,
- button rendering keys currently depend on `label`,
- there is no explicit typed contract for prompt identifiers,
- there is no feature-specific verification plan covering the shared send path.

## Locked Decisions

### Prompt Ownership

- Prompt templates live only in `src/constants/simulationQuickPrompts.ts`.
- UI components consume prompt metadata and never define prompt text inline.

### Prompt Transport

- Quick prompts are sent through the same `handleSend(content)` flow as manual textarea messages.
- No dedicated API endpoint, no alternate payload shape, and no special backend flag are introduced.

### Prompt Set

The feature includes exactly 3 prompts:

- physical exam,
- laboratory diagnostics,
- instrumental diagnostics.

If the product needs more prompts later, they must extend the same typed config shape instead of
adding ad hoc button definitions in the component.

## Tasks

### F1.1 Add Prompt Constants In `src/constants/simulationQuickPrompts.ts`

Replace the current loose array with a typed source of truth.

#### Required changes

- Introduce a stable prompt key union:
  - `physical-exam`
  - `lab-diagnostics`
  - `instrumental-diagnostics`
- Define a typed config shape, for example:
  - `key`
  - `label`
  - `content`
- Export the prompts as a readonly constant collection.
- Optionally export:
  - `SimulationQuickPromptKey`
  - `SimulationQuickPrompt`
  if needed by consuming components.

#### Content requirements

Replace the placeholder text with the locked templates from `PLAN.md`:

- physical exam prompt,
- laboratory diagnostics prompt,
- instrumental diagnostics prompt.

#### Rules

- Keep strings ASCII-only.
- Keep labels concise for button UI.
- Prefer `as const` or a narrow typed array/object so consumers get literal key inference.
- Do not duplicate prompt text anywhere else in the codebase.

Goal: one canonical prompt config with stable identifiers and correct final prompt text.

---

### F1.2 Render Buttons From Typed Config

Keep `ChatInputBar.vue` presentation-focused and fully config-driven.

#### Required changes

- Update `ChatInputBar.vue` prop typing to consume the final prompt type from constants.
- Use `prompt.key` as the `v-for` key instead of `prompt.label`.
- Continue rendering buttons from `quickPrompts` passed by the parent.
- Keep the component dumb:
  - no inline prompt text,
  - no switch/case by label,
  - no feature logic about which prompts should exist.

#### Interaction rules

- Quick prompt buttons respect the same disabled state as the text input/send button.
- Buttons must be non-interactive when:
  - chat input is disabled,
  - a send request is pending.
- Clicking a button emits only the resolved prompt content upward.

#### Optional cleanup

- If useful for clarity, rename local types/imports so it is obvious these are simulation quick prompts,
  not generic button presets.

Goal: button rendering is purely declarative and stable even if labels change later.

---

### F1.3 Wire Button Click To Message Send

Verify and, if necessary, tighten the shared send path so quick prompts are indistinguishable
from manual messages at the transport layer.

#### Required behavior

- `ChatInputBar` emits `send(prompt.content)`.
- `ChatView.vue` routes that event into the existing `handleSend(content)` method.
- `handleSend(content)` calls `simulationApi.sendMessage(sessionId, content)`.
- All existing pending, conflict, and streaming logic remains shared.

#### Validation points

- No duplicate quick-prompt-specific API method is introduced.
- Quick prompts reuse:
  - `canSendMessage`,
  - `isSendPending`,
  - conflict handling for `409`,
  - streaming startup via socket connect.
- Prompt send must clear/set the same transient UI state as manual sends:
  - `pendingSentMessage`,
  - `isStreamingActive`,
  - `streamingContent`.

#### UX expectations

- A quick prompt click should produce the same visible doctor-message pending state as typed input.
- If sending is blocked by backend/session state, the user sees the same inline conflict/error messaging
  already used for manual messages.

Goal: quick prompts are shortcuts, not a parallel message system.

## Definition Of Done

- `src/constants/simulationQuickPrompts.ts` contains the final 3 prompts with:
  - stable typed keys,
  - concise labels,
  - full prompt text from `PLAN.md`.
- `ChatInputBar.vue` renders prompt buttons using typed config and `prompt.key`.
- `ChatInputBar.vue` contains no hardcoded quick-prompt strings.
- `ChatView.vue` passes the shared prompt config into `ChatInputBar`.
- Quick prompt clicks use the existing `handleSend()` flow with no separate API path.
- Quick prompts respect disabled/pending/message-in-flight state exactly like manual sends.
- No `any` is introduced.
- ASCII-only source is preserved.

## Verification

### Manual checks

- Open a chat session in `IN_PROGRESS` with `OPENING_READY`.
- Confirm 3 quick prompt buttons render above the textarea.
- Click each button and verify:
  - one REST message request is sent through the normal send flow,
  - the pending doctor message appears,
  - patient reply streaming starts normally.
- Confirm buttons do nothing while:
  - opening is not ready,
  - a reply is already streaming,
  - a send request is pending.
- Confirm conflict/error messaging matches manual send behavior.

### Static checks

- `bun run type-check`
- `bun run el:check`
- `bun run sl:check`

## Risks And Mitigations

**Risk: labels are used as identity and become unstable if copy changes.**  
Mitigation: use explicit typed `key` values for render keys and future analytics/test hooks.

**Risk: prompt text drifts from the product spec.**  
Mitigation: copy the locked prompt templates from `PLAN.md` exactly into the constants file and keep that
file as the only source of truth.

**Risk: quick prompts silently bypass shared send guards later.**  
Mitigation: keep the feature wired to `handleSend(content)` only; do not add a separate click handler in
`ChatView.vue` that reimplements send logic.

## Execution Order

1. Finalize `src/constants/simulationQuickPrompts.ts` with typed keys and final prompt text.
2. Update `src/components/chat/ChatInputBar.vue` to consume the typed config and use stable render keys.
3. Verify `src/views/ChatView.vue` continues to pass the config through the shared `handleSend()` path.
4. Run type/lint checks and manual chat-flow verification.
